# What is Version Control?

1. What is Version Control

  • A component of software configuration management, version control, also known as revision control or source control, is the management of changes to documents, computer programs, large web sites, and other collections of information
  • Three of the most popular version control systems are:
    • Git
    • Subversion
    • Mercurial
  • There are two main types of version control system models:
    • the centralized model all users connect to a central, master repository
    • the distributed model each user has the entire repository on their computer
  • Git is the version control tool and GitHub is the service that hosts Git projects
  • Version control systems aim to solve a specific problem that programmers face: "storing and sharing multiple versions of code files"

3. Git and Version Control Terminology

  • Version Control System / Source Code Manager: VCS or SCM is a tool that manages different versions of source code
  • Commit: Git thinks of its data like a set of snapshots of a mini filesystem, everytime you commit (save the state of your project in Git), it basically takes a picture of what all your files look like at that moment and stores a reference to that snapshot
    • Think of it as a save point in a game - it saves your project's files and any information about them
    • Everything you do in Git is to help you make commits, so a commit is the fundamental unit in Git
    • When a commit is made, only the changes that are in the Staging Index are saved in the repository
  • Repository / repo: a directory which contains your project work, as well as a few files (hidden by default on Mac OS X) which are used to communicate with Git
    • Repos can exist either locally on your computer or as a remote copy on another computer
    • A repository is made up of commits
  • Working Directory: is the files that you see in your computer's file system
    • When you open your project files up on a code editor, you're working with files in the working directory
    • This is in contrast to the files that have been saved (in commits) in the repository
  • Checkout: is when content in the repository has been copied to the Working Directory
  • SHA: is basically an ID number for each commit, "SHA" is shorthand for "Secure Hash Algorithm"
  • Branch: is when a new line of development is created that diverges from the main line of development, this alternative line of development can continue without altering the main line
  • The key terms can be found here

# Create a Git Repo

1. Intro

  • git init is used to create brand new repositories on your computer
  • git clone is used to copy existing repos from somewhere else to your local computer
  • git status is used to check the status of a repo

2. Create a Repo From Scratch

  • Running the git init command sets up all the necessary files and directories that Git will use to keep track of everything, all these files are stored in a directory called .git (notice the . at the beginning - that means it'll be a hidden directory on Mac/Linux)
  • Brief synopsis on each of the items in the .git directory:
    • config file: where all project specific configuration settings are stored
    • description file: this file is only used by the GitWeb program, so we can ignore it
    • hooks directory: this is where we could place client-side or server-side scripts that we can use to hook into Git's different lifecycle events
    • info directory: contains the global exclues file
    • objects directory: this directory will store all of the commits we make
    • refs directory: this directory holds pointers to commits (basically the "branches" and "tags")

3. Clone An Existing Repo

  • $ git clone <path-to-repository-to-clone>
    • takes the path to an existing repo
    • by default will create a directory with the same name as the repo that's being cloned
    • can be given a second argument that will be used as the name of the directory
    • will create the new repo inside of the current working directory

4. Determine A Repo's Status

  • git status command will display a lot of information depending on the state of your files, the working directory, and the repository

# Review a Repo's History

1. Intro

  • git log displays information about the existing commits
  • git show displays information about the given commit

2. Displaying A Repository's Commits

  • Always run the git status command, especially when returning to a project after a period of time
  • to scroll down, press
    • j or down arrow key to move down one line at a time
    • d to move by half the page screen
    • f to move by a whole page screen
  • to scroll up, press
    • k or up arrow key to move up one line at a time
    • u to move by half the page screen
    • b to move by a whole page screen
  • press q to quit out of the log (returns tot he regular command prompt)
  • By default, the git log command displays: the SHA, author, date, and message of every commit in the repository

3. Changing How Git Log Displays Information

  • $ git log --oneline
    • The --oneline flag shows just the short SHA and the commit message
    • lists one commit per line
    • shows the first 7 characters of the commit's SHA
    • shows the commit's message

4. Viewing Modified Files

  • $ git log --stat
    • displays the file(s) that have been modified
    • displays the number of lines that have been added/removed
    • displays a summary line with the total number of modified files and lines that have been added/removed

5. Viewing File Changes

  • $ git log -p
    • The -p flag (which is the same as the --patch flag) is used to alter how git log displays information
    • displays the files that have been modified
    • displays the location of the lines that have been added/removed
    • displays the actual changes that have been made
  • both git log -p --stat and git log --stat -p displays with the stats info above the patch info
  • the -w flag ignores the whitespace changes in the patch information

6. Viewing A Specific Commit

  • By supplying a SHA, the git log -p command will start at that commit
  • $ git show
    • Displays only the most recent commit
    • Typically, a SHA is provided as a final argument
    • The output of the 'git show' command is exactly the same as the 'git log -p' command
    • By default the 'git show' displays: the commit, author, date, commit message, patch information
  • 'git show' can be combined with most of the other flags we've looked at:
    • '--stat' - to show the how many files were changed and the number of lines that were added/removed
    • '-p' or '--patch' - this the default, but if '--stat' is used, the patch won't display, so pass '-p' to add it again
    • '-w' - to ignore changes to whitespace

# Add Commits to a Repo

1. Intro

  • git add adds files from the working directory to the staging index
  • git commit takes files from the staging index and save them in the repository
  • git diff displays the difference between two versions of a file

2. Git Add

  • Running git init multiple times doesn't cause any problems since it just re-initializes the Git directory
  • Use git rm --cached <file> to unstage
    • It is not like the shell's rmcommand
    • git rm --cached <file> will not destroy any of your work; it just removes it from the Staging Index
  • $ git add .
    • The period refers to the current directory and can be used as a shortcut to refer to all files and directories (including all nested files and directories)
    • Becareful to not accidentally include more files than you meant to
  • $ git add <file1> <file2> ... <fileN>
    • takes a space-separated list of file names
    • alternatively, the period can be used in place of a list of files to tell Git to add the current directory (and all nested files)

3. Git Commit

  • git commit command takes files from the Staging Index and saves them in the repository
  • $ git commit -m "Initial commit"
    • Use the -m flag to pass your message directly on the command line
  • The best way to think about what should be in a commit is to think, "What if all changes introduced in this commit were erased?" if a commit were erased, it should only remove one thing

4. Commit Messages

  • Some important things to think about when crafting a good commit message:
    • do keep the message short (less than 60-ish characters
    • do explain what the commit does (not how or why!)
    • do not explain why the changes are made
    • do not explain how the changes are made (that's what git log -p is for!)
    • do not use the word "and", if you have to use "and", your commit message is probably doing too many changes - break the changes into separate commits
  • The best way to come up with a commit message is to finish this phrase, "This commit will..."
  • Above all, be consistent in how you write your commit messages!
  • If you need to explain why a commit needs to be made, you can after the first line in the commit message
  • After the message, leave a blank line, and then type out the body or explanation including details about why the commit is needed

5. Git Diff

  • $ git diff
  • The command is used to see changes that have been made but haven't been committed, yet
  • This command displays: the files that have been modified, the location of the lines that have been added/removed, the actual changes that have been made

6. Having Git Ignore Files

  • The .gitignore file is used to tell Git about the files that Git should not track
  • This file should be placed in the same directory that the .git directory is in
  • Globbing lets you use special characters to match patterns/characters, in the .gitignore file, you can use the following:
    • blank lines can be used for spacing
    • # - marks line as a comment
    • * - matches 0 or more characters
    • ? - matches 1 character
    • [abc] - matches a, b, or c
    • ** - matches nested directories
  • Create useful .gitignore files for you project here

# Tagging, Branching, and Merging

1. Intro

  • git tag adds tags to specific commits
  • git branch allows multiple lines of development
  • git checkout switch between different branches and tags
  • git merge combines changes from different branches

2. Tagging

  • Tags let you point out particular commits to make them stand out from others
  • $ git tag -a beta
    • Adds a tag to the most recent commit
    • Adds a tag to a specific commit if a SHA is passed
    • -a flag is used to create an annotated tag, recommended becausee it includes more information such as: the person who made the tag, the date the tag was made, and a message for the tag
  • A Git tag can be deleted with the -d flag (for delete!) and the name of the tag, $ git tag -d v1.0

3. Branching

  • $ git branch
    • can be used to list all branch names in the repository, to create new branches, and to delete branches
  • To create a branch, all you have to do is use git branch and provide it the name of the branch you want to create
  • When a commit is made it will be added to the current branch
  • $ git checkout <branch>
    • Running this command will remove all files and directories from the Working Directory that Git is tracking (files that Git tracks are stored in the repo, so nothing is lost)
    • It will go into the repository and pull out all of the files and directories of the commit that the branch points to
  • $ git branch alt-sidebar-loc 42a69f
    • This will create the alt-sidebar-loc branch and have it point to the commit with SHA 42a69f
  • After a branch's changes have been merged, you probably won't need the branch anymore, if you want to delete the branch use the -d flag
  • $ git branch -d sidebar
    • The command deletes the provided branch, in this case, the "sidebar" branch
    • You cannot delete a branch that you're currently on
    • Git won't let you delete a branch if it has commits on it that aren't on any other branch (meaning the commits are unique to the branch that's about to be deleted)
    • To force deletion, use a capital -D flag
  • $ git checkout -b <branch> can be used to switch to a newly created branch
  • Further Research
    • Git Branching - Basic Branching and Merging from the Git Docs
    • Learn Git Branching
    • Git Branching Tutorial from the Atlassian Blog

4. Branching Effectively

  • $ git log --oneline -- decorate --graph --all
    • Running this command will show all branches and commits in the repository
    • --graph flag adds the bullets and lines to the leftmost part of the output, this shows the actual branching that's happening
    • --all flag is what displays all of the branches in the repository

5. Merging

  • Combining branches together is called merging, there are two main types of merges in Git, a regular merge and a Fast-forward merge
  • Very important to know which branch you're on when you're about to merge branches together, remember that making a merge makes a commit ⚠️
  • $ git merge <name-of-branch-to-merge-in>
  • When we merge, we're merging some other branch into the current(checked-out) branch, not merging two branches into a new branch
  • A default message is supplied for a merge commit
  • Whichever branch the special HEAD pointer is pointing at, that's the branch that will have the merge commit

6. Merge Conflicts

  • A merge conflict happens when a merge fails
  • Git tracks lines in files, a merge conflict will happen when the exact same line(s) are changed in separate branches
  • So the same line on two or more different branches have to be edited... and then try to merge them
  • The editor has the following merge conflict indicators:
    • <<<<<<< HEAD everything below this line (until the next indicator) shows you what's on the current branch
    • ||||||| merged common ancestors everything below this line (until the next indicator) shows you what the original lines were
    • ======= is the end of the original lines, everything that follows (until the next indicator) is what's on the branch that's being merged in
    • >>>>>>> branch-name is the ending indicator of what's on the branch that's being merged in
  • To resolve a merge conflict, first choose which line(s) to keep and, then, remove all lines with indicators

# Undoing Changes

1. Intro

  • git commit --amend alters the most-recent commit
  • git revert reverses given commit
  • git reset erases commits

2. Modifying The Last Commit

  • $ git commit --amend
    • If Working Directory is clean (meaning there are not any uncommitted changes in the repository), then running the command will let you provide a new commit message
    • Alternatively, the command will let you include files (or changes to files) you might have forgotten to include
      • edit the file(s)
      • save the file(s)
      • stage the file(s)
      • and run git commit --amend
      • This updates the most-recent commit instead of creating a new one

3. Reverting A Commit

  • $ git revert <SHA-of-commit-to-revert>
    • Will undo the changes that were made by the provided commit
    • creates a new commit to record the changes

4. Resetting Commits

  • At first glance, resetting might seem coincidentally close to reverting, but they are actually quite different
  • Reverting creates a new commit that reverts or undos a previous commit
  • Resetting, on the other hand, erases commits
  • Becareful with Git's resetting capabilities, this is one of the few commands that lets you erase commits from the repository ⚠️
  • If a commit is no longer in the repo, then its content is gone
  • There are special characters called "Ancestry References" that tell Git about relative references to another commit:
    • ^ - indicates the parent commit
    • ~ - indicates the first parent commit
  • The following indicate the parent commit of the current commit
    • HEAD^
    • HEAD~
    • HEAD~1
  • The following indicate the grandparent commit of the current commit
    • HEAD^^
    • HEAD~2
  • The following indcate the great-grandparent commit of the current commit
    • HEAD^^^
    • HEAD~3
  • The main difference between the ^ and he ~ is when a commit is created from a merge
  • A merge commit has two parents, the first parent is the branch you were on when you ran git merge while the second parent is the branch that was merged in
  • The ^ reference is used to indicate the first parent of the commit while ^2 indicates the second parent
  • $ git reset <reference-to-commit>
    • Move the HEAD and current branch pointer to the referenced commit
    • Erase commits
    • Move committed changes to the staging index
    • Unstage committed changes
  • The way that Git determines if it erases, stages previously committed changes, or unstages previously committed changes is by the flag that is used
    • --mixed moves the content to the Working Directory (--mixed is the default)
    • --soft moves the content to the Staging Index
    • --hard moves the content to the trash

# Working with Remotes

1. Intro

  • Watched video clip

2. Remote Repositories

  • When collaborating with other developers make sure to create a new branch that has a descriptive name that describes what changes it contains
  • git remote manage remote repo
  • git push send changes to the remote
  • git pull retrieve updates from the remote
  • A local repository is the one that you work on in your local machine, while a remote repositories live elsewhere (e.g. a friend's computer, GitHub, etc.)

3. Add A Remote Repository

  • The word "origin" is the defacto name that's used to refer to the main remote repository, it is possible to rename this to something else, but typically it's left as "origin"
  • $ git remote -v
    • Use this command with the -v flag if you want to see the full path to the remote repository
  • Git is a version control tool
  • GitHub is a service to host Git projects
  • $ git remote add is used to add a connection to a new remote repository

4. Push Changes To A Remote

  • $ git push origin master
    • The command takes the shortname of the remote repository you want to send commits to and the name of the branch that has the commits you want to send

5. Pulling Changes From A Remote

  • $ git pull origin master
  • If there are changes in a remote repo that you'd like to include in your local repository, then you wan to pull in those changes
  • When git pull is run, the following things happen:
    • the commit(s) on the remote branch are copied to the local repository
    • the local tracking branch (origin/master) is moved to point to the most recent commit
    • the local tracking branch (origin/master) is merged into the local branch (master)
  • Changes can be manually added on GitHub (but this is not recommended, so don't do it)

6. Pull vs Fetch

  • You can think of the git pull command as doing two things:
    • fetching remote changes (which adds the commits to the local repository and moves the tracking branch to point to them)
    • merging the local branch with the tracking branch
  • The git fetch command is just the first step, it just retrieves the commits and moves the tracking branch, it does not merge the local branch with the tracking branch
  • $ git fetch origin master
  • the same information provided to git pull is passed to git fetch:
    • the shortname of the remote repository
    • the branch with commits to retrieve

7. Lesson Wrap Up

  • git remote to manage a remote repository
  • git push to send changes
  • git fetch to retrieve updates
  • git pull to retrieve and automatically merge updates

# Working on Another Developer's Repository

1. Intro

  • Learning objectives: Forking a repository, reviewing another developer's changes, and knowing what changes to make

2. Forking a Repository

  • Forking is an action that's done on a hosting service, like GitHub
  • Forking a repository creates an identical copy of the original repository and moves this copy to your account
  • You have total control over this forked repo
  • Modifying your forked repository does not alter the original repository in any way

3. Forking a Repository

  • $ git shortlog
    • The command displays an alphabetical list of names and the commit messages that go along with them
  • $ git shortlog -s -n
    • -s flag to show just the number of commits (rather than each commit's message)
    • -n flag to short them numerically (rather than alphabetically by author name)
  • $ git log --author=<name>
    • Use quotes for name if needed, otherwise it will cause an error
  • git log --grep=<key word>
    • Filters down to just the commits that reference the key word(s)

4. Determining What To Work On

  • Before you start doing any work, make sure to look for the project's CONTRIBUTING.md file
  • Look at GitHub issues for the project
    • look at the existing issues to see if one is similar to the change you want to contribute
    • if necessary create a new issue
    • communicate the changes you'd like to make to the project maintainer in the issue
  • When you start developing, commit all of your work on a topic branch:
    • do no work on the master branch
    • make sure to give the topic branch clear, descriptive name
  • As a general best practices include:
    • Write descriptive commit messages
    • Create small, focused commits
    • Update the README

# Staying in Sync with a Remote Repository

1. Intro

  • Watched video clip

2. Create a Pull Request

  • A pull request is a request for the source repository to pull in your commits and merge them with their project
  • To create a pull request, a couple of things need to happen:
    • you must fork the source repository
    • clone your fork down to your machine
    • make some commits (ideally on a topic branch!)
    • push the commits back to your fork
    • create a new pull request and choose the branch that has your new commits

3. Stay in sync with source project

  • The word upstream is typically used to refer to the source repository
  • When working with a project that you've forked, the original project's maintainer may continue adding changes to their project, you will want to keep your fork of their project in sync with theirs so that you can include any changes they make
  • To get commits from a source repository into your forked repository on GitHub you need to:
    • get the cloneable URL of the source repository
    • create a new remote with the git remote add command
      • use the shortname upstream to point to the source repository
      • provide the URL of the source repository
    • fetch the new upstream remote
    • merge the upstream's branch into a local branch
    • push the newly updated local branch to your origin repo
  • $ git remote rename <current name> <new name>

4. Manage an active PR

  • Working on an active pull request is mostly about communication
  • If the project's maintainer is requesting changes to the pull request, then:
    • make any necessary commits on the same branch in your local repository that your pull request is based on
    • push the branch to your fork of the source repository
  • The commits will then show up on the pull request page

5. Squash Commits

  • git rebase command is used to squash commits together
  • git rebase -i HEAD~3
    • The -i flag stands for "interactive", you can perform a rebase in a non-interactive mode, however, it is recommended to do interactive rebasing while learning to rebase
